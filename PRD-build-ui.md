# PRD: /build-ui スキル — Generator/Evaluator UI構築ループ

## Problem Statement

Claude Code で UI を作成・改善する際、エージェントが自分の出力を自分で評価すると「自己評価バイアス」が発生し、平凡な成果物を高評価してしまう。特にデザインのような主観的タスクでは、ソフトウェアテストのようなバイナリチェックが存在しないため、品質の上限が低い。また、単一エージェントではコンテキストウィンドウが埋まるにつれて出力の一貫性が失われ、長時間の反復的改善が困難になる。

## Solution

Anthropic 記事「Harness Design for Long-Running Apps」の Generator/Evaluator アーキテクチャを Claude Code スキルとして実装する。`/build-ui` スキルは Planner・Generator・Evaluator の3エージェントを Agent ツールでサブエージェントとして起動し、ファイルベースのハンドオフで通信しながら 3〜5 サイクルの反復ループで UI を構築・改善する。

Evaluator は Playwright MCP でライブページを操作・スクリーンショット取得し、4つの評価基準（デザイン品質・独自性・技巧・機能性）に基づく批評を生成する。Generator はそのフィードバックを受けて改善するか、美的方針を全面的に転換するかを判断する。

## User Stories

1. 開発者として、UIの作成を指示したら自動的に反復改善ループが回ってほしい。手動でフィードバックを繰り返す手間を省くため
2. 開発者として、Evaluator がブラウザで実際の表示を確認してほしい。コードだけでは見えないレイアウト崩れや視覚的問題を検出するため
3. 開発者として、Evaluator がスクリーンショットを取得して評価してほしい。視覚的なデザイン品質を客観的に判断するため
4. 開発者として、Generator と Evaluator が分離されていてほしい。自己評価バイアスを排除し、独立した批判的フィードバックを得るため
5. 開発者として、Planner が要件から具体的な仕様を展開してほしい。曖昧な指示からでも明確な実装方針が得られるため
6. 開発者として、Generator と Evaluator がスプリント契約で事前に合意してほしい。何を作り何で検証するかのズレを防ぐため
7. 開発者として、エージェント間の通信がファイルベースで行われてほしい。コンテキストウィンドウの肥大化を防ぎ、各エージェントが新鮮な状態で動作するため
8. 開発者として、反復回数を 3〜5 サイクルに制限してほしい。コストと時間を予測可能に保つため
9. 開発者として、既存の Next.js/React アプリの UI を改善するのにこのスキルを使いたい。既存コードベースの文脈を理解した上で改善するため
10. 開発者として、新規の UI ページをゼロから作成するのにもこのスキルを使いたい。プロンプトから高品質なページを生成するため
11. 開発者として、デザイン品質と独自性が重視された評価を受けたい。テンプレート的な出力ではなく、独自の視覚的アイデンティティを持つ UI を作るため
12. 開発者として、このスキルがフレームワーク非依存で動作してほしい。Next.js 以外のプロジェクトでも使えるようにするため
13. 開発者として、各サイクルの評価結果をファイルで確認したい。改善の過程をトレースし、どこで品質が上がったかを把握するため
14. 開発者として、Evaluator が機能性（ユーザビリティ）も検証してほしい。見た目だけでなく、実際に使えるかどうかも確認するため
15. 開発者として、Generator が評価フィードバックに基づいて方針転換できてほしい。漸進的改善だけでなく、根本的な再設計も可能にするため
16. 開発者として、Chrome 上でブラウザが動く様子を目視で確認したい。エージェントの操作過程を自分の目でも追えるようにするため

## Implementation Decisions

### アーキテクチャ

GAN に着想を得た3エージェント分離構成：

| エージェント | 役割 | 使用ツール |
|-------------|------|-----------|
| **Planner** | ユーザーの要件を包括的な仕様に展開。デザイン方針・技術制約・評価基準の重み付けを決定 | Explore エージェント、コードベース読み取り |
| **Generator** | 仕様に基づいてコードを実装。Evaluator のフィードバックを受けて改善または方針転換 | Edit、Write、Bash（dev server 確認） |
| **Evaluator** | Playwright MCP でライブページを操作し、4基準で採点・批評を生成 | Playwright MCP（browser_navigate, browser_screenshot, browser_snapshot, browser_click）、Chrome DevTools MCP（list_console_messages, list_network_requests） |

### スプリント契約メカニズム

各反復サイクルの前に Generator と Evaluator がファイルベースで契約を交渉する：

1. Generator が「何を構築するか」「成功をどう検証するか」を提案（契約ファイルに書き出し）
2. Evaluator がレビューし、合意または修正を要求
3. 合意後、Generator が実装を開始

### 評価基準（記事の4基準を採用）

| 基準 | 重み | 内容 |
|------|------|------|
| **デザイン品質** | 高 | 一貫性のある全体像、独自のムード、色・タイポグラフィ・レイアウト・画像の効果的な組み合わせ |
| **独自性** | 高 | カスタムな設計判断、テンプレートレイアウトやライブラリデフォルトの回避 |
| **技巧（Craft）** | 中 | タイポグラフィ階層、余白、色調の調和、コントラスト比 |
| **機能性** | 中 | ユーザビリティ：ユーザーがUIを理解し、主要アクションを見つけ、タスクを完了できるか |

各基準に合格しきい値を設け、不合格の場合は詳細なフィードバックとともに Generator に差し戻す。

### ファイルベースのハンドオフプロトコル

プロジェクト内に `.build-ui/` ディレクトリを作成し、以下のファイルで通信：

| ファイル | 用途 |
|---------|------|
| `spec.md` | Planner が作成する仕様書 |
| `contract.md` | Generator/Evaluator のスプリント契約 |
| `evaluation-{n}.md` | 各サイクルの Evaluator フィードバック（スコア + 批評） |
| `screenshots/` | Evaluator が取得したスクリーンショット |

Generator は `evaluation-{n}.md` を読んで次のサイクルの改善方針を決定する。Evaluator は前回の自分の評価は読まず、常に新鮮な目で評価する。

### 反復ループの制御

- デフォルト 3〜5 サイクル
- 全基準が合格しきい値を超えた場合は早期終了
- 各サイクルで Generator は「改善」か「方針転換」かを戦略的に判断

### スキルのモジュール構成

`frontend_dev_tools` リポジトリ内に配置：

| ファイル | 役割 |
|---------|------|
| `skills/build-ui/SKILL.md` | メインオーケストレータ。ループ制御、Agent ツール呼び出し |
| `skills/build-ui/PLANNER.md` | Planner エージェントへのプロンプト |
| `skills/build-ui/GENERATOR.md` | Generator エージェントへのプロンプト |
| `skills/build-ui/EVALUATOR.md` | Evaluator エージェントへのプロンプト（評価基準・採点ルーブリック含む） |
| `skills/build-ui/HANDOFF.md` | ファイルベースハンドオフのプロトコル定義 |

### フレームワーク非依存

- スキル自体は特定フレームワークに依存しない
- Planner が対象プロジェクトの技術スタックを自動検出し、仕様に反映
- Evaluator は Playwright MCP 経由でブラウザを操作するため、技術スタックに関係なく動作

## Testing Decisions

自動テストの対象ではないため（成果物はプロンプトファイル）、手動検証シナリオで品質を担保する。

### 検証シナリオ

`slide_sample_choose/webapp`（Next.js 16 + React 19）で以下を実行：

**シナリオ 1: 既存 UI の改善**
1. `/build-ui` を実行し、「トップページのデザインを改善して」と指示
2. Planner が仕様を作成することを確認
3. Generator/Evaluator のスプリント契約が生成されることを確認
4. Evaluator がブラウザでスクリーンショットを取得し、4基準で採点することを確認
5. 3〜5 サイクルのループが回ることを確認
6. `.build-ui/` ディレクトリに評価ファイルが蓄積されることを確認

**シナリオ 2: 新規ページ作成**
1. `/build-ui` を実行し、「設定ページを新規作成して」と指示
2. ゼロからページが生成され、反復的に改善されることを確認

### 良いテストの基準

- エージェント間のファイルハンドオフが正しく行われているか（外部動作）
- 各サイクルの評価スコアが記録されているか
- Evaluator が実際に Playwright MCP ツールを使用しているか

## Out of Scope

- **他のスキル（/check-ui, /debug-frontend, /gen-e2e）の実装** — 別途 Phase 3 で対応
- **Claude Agent SDK によるオーケストレーション** — Claude Code スキルとして実装
- **CI/CD 統合** — ローカル開発環境での利用に限定
- **評価基準のカスタマイズUI** — 記事の4基準を固定で使用
- **複数ページにまたがるワークフロー** — 単一ページ/コンポーネントの改善に集中
- **自動デプロイ** — コード生成・改善まで

## Further Notes

- 記事によると、評価基準の文言自体が Generator の出力を方向付ける効果がある（「プロンプトの波及効果」）。基準を慎重に設計することで、評価前の初回出力から品質が向上する
- Evaluator のプロンプトは反復的にチューニングが必要。初期状態では「正当な問題を見つけつつ重要でないと合理化してしまう」傾向がある。実運用しながら調整する
- Opus 4.6 のコンテキスト維持能力を活かし、記事で言及されている「スプリント分割」は不要。ただし Generator/Evaluator の分離は自己評価バイアス排除のために維持する
- コストは 1 回の実行で推定 $5〜20（3〜5 サイクル）。記事の $200/6時間と比較して大幅に抑えている
