# ファイルベースハンドオフプロトコル

エージェント間の通信はすべてファイル経由で行う。プロジェクトルートに `.build-ui/` ディレクトリを作成し、以下の構造で管理する。

## ディレクトリ構造

```
.build-ui/
├── spec.md                  # Planner → Generator/Evaluator
├── contract.md              # Generator ↔ Evaluator
├── evaluation-1.md          # Evaluator → Generator (サイクル1)
├── evaluation-2.md          # Evaluator → Generator (サイクル2)
├── evaluation-{n}.md        # Evaluator → Generator (サイクルn)
└── screenshots/
    ├── cycle-1-initial.png
    ├── cycle-1-detail-1.png
    ├── cycle-2-initial.png
    └── ...
```

## spec.md フォーマット

Planner が作成する仕様書。Generator と Evaluator の両方が参照する。

```markdown
# UI仕様書

## 概要
<!-- ユーザーの要件を1〜3文で要約 -->

## 対象
- **タイプ**: 新規作成 / 既存改善
- **対象ファイル**: <!-- 既存改善の場合のみ。対象ファイルパス -->

## 技術スタック
- **フレームワーク**: <!-- 自動検出結果 -->
- **スタイリング**: <!-- Tailwind / CSS Modules / etc. -->
- **主要ライブラリ**: <!-- 関連ライブラリ -->

## デザイン方針
- **全体のムード**: <!-- モダン、ミニマル、遊び心 等 -->
- **カラーパレット**: <!-- 既存の色体系 or 提案 -->
- **タイポグラフィ**: <!-- フォント、階層 -->
- **レイアウト方針**: <!-- グリッド、フレックス、制約 -->

## 機能要件
<!-- 番号付きリスト -->
1. ...
2. ...

## 評価基準の重み付け
| 基準 | 重み | 備考 |
|------|------|------|
| デザイン品質 | 高 | <!-- プロジェクト固有の注記 --> |
| 独自性 | 高 | |
| 技巧 | 中 | |
| 機能性 | 中 | |
```

## contract.md フォーマット

Generator と Evaluator が各スプリントの前に合意する契約。

```markdown
# スプリント契約

## サイクル: {n}

## 構築内容
<!-- Generator が今回のサイクルで何を作る/変更するか -->
1. ...
2. ...

## 検証方法
<!-- Evaluator がどのようにテストするか -->
1. ...
2. ...

## 合意ステータス
- Generator: 合意 / 未合意
- Evaluator: 合意 / 未合意

## 修正履歴
<!-- 交渉で変更された内容があれば記録 -->
```

## evaluation-{n}.md フォーマット

Evaluator が各サイクル後に書き出すフィードバック。Generator はこれを読んで次の改善方針を決定する。

```markdown
# 評価レポート — サイクル {n}

## スコア

| 基準 | スコア (1-10) | 重み | 合格しきい値 | 判定 |
|------|-------------|------|------------|------|
| デザイン品質 | {score} | 高 | 7 | PASS/FAIL |
| 独自性 | {score} | 高 | 7 | PASS/FAIL |
| 技巧 | {score} | 中 | 6 | PASS/FAIL |
| 機能性 | {score} | 中 | 6 | PASS/FAIL |

**総合判定**: PASS / FAIL

## デザイン品質の評価
<!-- 一貫性、ムード、色・タイポグラフィ・レイアウトの組み合わせ -->

## 独自性の評価
<!-- テンプレート回避、カスタム設計判断 -->

## 技巧の評価
<!-- タイポグラフィ階層、余白、色調の調和、コントラスト比 -->

## 機能性の評価
<!-- ユーザビリティ、主要アクションの発見しやすさ、タスク完了可能性 -->

## 致命的な問題
<!-- レイアウト崩れ、JSエラー、リンク切れ等 -->

## 改善提案
<!-- 次のサイクルで Generator が取り組むべき具体的な改善点 -->
1. ...
2. ...

## 推奨アプローチ
- [ ] 漸進的改善（現在の方向性を維持して磨く）
- [ ] 方針転換（根本的に異なるアプローチを試す）
<!-- どちらかにチェックを入れ、理由を記述 -->
```

## スクリーンショット命名規則

```
screenshots/cycle-{n}-initial.png      # ページ全体のスクリーンショット
screenshots/cycle-{n}-detail-{m}.png   # 注目箇所の詳細スクリーンショット
screenshots/cycle-{n}-mobile.png       # モバイル表示（必要な場合）
```

## ルール

1. **書き込み権限**: 各ファイルは担当エージェントのみが書き込む
   - `spec.md`: Planner のみ
   - `contract.md`: Generator と Evaluator が交互に更新
   - `evaluation-{n}.md`: Evaluator のみ
2. **読み取り**: どのエージェントも全ファイルを読める
3. **Evaluator の独立性**: Evaluator は過去の `evaluation-{n-1}.md` を読まない。常に新鮮な目で評価する
4. **上書き禁止**: `evaluation-{n}.md` は一度書いたら変更しない。新しいサイクルは新しいファイル
